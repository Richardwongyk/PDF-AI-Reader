# PDF AI 阅读器 · 极简设计报告

## 产品目标

理工科学生/研究者读英文论文用的桌面工具。核心目标：

1. **PDF 阅读**：快速打开、滚动、跳页、缩放，长文档不被后台任务拖慢。
2. **段落翻译**：双击段落在原文下方显示 AI 翻译，公式和术语保持可控。
3. **全文问答**：问题必须基于当前 PDF 的检索证据和上下文回答。
4. **公式证据链**：born-digital PDF 先读文本层、glyph、font、bbox、vector；图片/扫描/低置信才走 OCR/MFR。
5. **审核写回**：低置信公式只做候选，accepted 后才允许增量进入知识库和 GraphRAG。

生成模型通过 LiteLLM 云端或 Ollama 本地后端路由；Ollama 是可选本地模式，不再把本地 4B 模型当成所有用户的默认前提。

## 三个核心流程

**打开 PDF**：PyMuPDF 读文本块、页面尺寸、glyph/font/bbox/vector/image 事实 → `DocumentChunker` 生成段落、行内公式候选和基础块 → `PdfViewer` 建立虚拟页面布局，只渲染可见页附近 → 后台构建 FTS/Chroma 知识库，GraphRAG 和公式多轮任务异步排队。

**翻译段落**：双击段落 → 插入 SplitWidget 裂缝容器 → 公式占位保护、术语表和上下文约束进入 prompt → LiteLLM/Ollama 流式返回 → QWebEngineView 渲染 Markdown + KaTeX。

**公式多轮**：r0 结构快扫不 OCR → r0.5 符号身份修复 → r1 缓存优先补救图片/扫描/needs_ocr → r2/r2a 写本地工具和 TinyBDMath 候选 → r3 云端语义复核只写 JSON → r4 写候选图谱 → r5 仅在 accepted 变化后增量 upsert 知识库并同步 accepted 图谱 artifact。

## 关键技术选型

| 组件 | 选型 | 原因 |
|------|------|------|
| GUI | PySide6 | Qt 桌面能力、信号/槽、QWebEngine 渲染 |
| PDF 引擎 | PyMuPDF / MuPDF | 快速渲染和文本层、glyph、bbox、vector 事实抽取 |
| 生成模型 | LiteLLM + Ollama | 云端/本地可配置路由，测试可降级 Mock |
| 知识库 | SQLite FTS5 / Chroma / LlamaIndex 编排 | 长文档快速全文召回，同时保留向量/RAG 扩展 |
| GraphRAG | SQLite artifact + 后续成熟图谱后端 | 先保证异步、可恢复、可追溯 |
| 公式视觉兜底 | Pix2Text / Paddle / MinerU / PEK worker | 只作为隔离候选后端，不进默认热路径 |
| 公式渲染 | KaTeX | 本地、轻量、适合裂缝交互 |

## 当前未完成边界

- 公式多轮、审核 UI、manual revision、evidence 预览、PDF bbox 定位和 r5 增量写回已经接线。
- Napkin 公式质量门禁仍失败，不能宣称 99.9% 公式还原。
- TinyBDMath 当前是 candidate-only，仍需要 hard label、decoder/verifier 和大样本质量证明。
- GraphRAG 当前有 artifact 管线和 accepted 同步，仍不是最终语义级图谱产品体验。
- 2026-05-28 性能修复尝试暴露 P0 阅读体验退化：极大缩放下快速滚动/跳页时，大页面
  tile-only 首帧可能无 fallback，用户看到黑底/空白页。下一步必须先保证滚动中间页始终有
  可辨识页面内容。
- 翻译公式保护仍需并发隔离：每个翻译请求应有独立公式占位 store，不能让流式翻译共享状态。

## 线程

| 线程/任务 | 职责 |
|------|------|
| 主线程 | Qt 事件循环、页面显示、交互 |
| 解析线程 | PDF 打开、分块、基础结构提取 |
| QThreadPool | 知识库构建、GraphRAG、公式索引小批任务 |
| 翻译/问答线程 | LLM 调用和流式回调 |
| 外部 worker | MinerU/Paddle/Pix2Text/PEK 等重工具隔离运行 |
